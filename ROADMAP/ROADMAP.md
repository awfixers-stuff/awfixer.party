# Awfixer Party — Engineering roadmap

This roadmap turns decisions in [`WORK.md`](./WORK.md) into phased work. It is a living document: adjust dates, owners, and scope as you learn.

**Convention:** Each **subphase** ends with a required task to **create or update an external TODO list** (GitHub Issues, Linear, Notion, etc.) with the **next** concrete items for that track, so execution does not stall at repo-boundary documentation.

---

## How to use this roadmap

1. Pick a **section** aligned with current risk (often **II + III** first: auth + data plane).
2. Execute **subphases** in order; each ends by refreshing your **external** tracker so the next engineer does not rely on markdown alone.
3. When architecture changes, update [`WORK.md`](./WORK.md) first, then this file, then external issues.

_Last line of every subphase checklist (repeated by design): **Create or update an external TODO list with the next set of items.**_
